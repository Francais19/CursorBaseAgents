# `.cursor`: как этим пользоваться, если вы только пришли в проект

Если коротко: папка `.cursor` нужна, чтобы Cursor работал по правилам проекта, понимал наш workflow и умел запускать готовые команды вроде `/implement`, `/review` и `/audit`.

Вам не нужно разбираться во всей структуре сразу. Для обычной работы достаточно понять:
- где писать команду
- какую команду выбрать
- что именно написать после команды

## Что сделать перед первым использованием

Если вы только подключили этот шаблон `.cursor` к новому проекту, сначала нужно его актуализировать под реальный код проекта.

Для этого используйте файл `.cursor/TEMPLATE_PROMPT_RU.md`.

Смысл простой: в шаблоне `.cursor` есть плейсхолдеры вроде `{{APP_FRAMEWORK}}`, `{{ORM}}`, `{{PACKAGE_MANAGER}}`, `{{INTEGRATIONS}}`. Перед первой нормальной работой их нужно заменить на реальные значения проекта.

Иначе Cursor может:
- опираться на шаблонные или пустые значения
- давать менее точные инструкции
- хуже понимать стек, тесты и интеграции проекта

Практический сценарий такой:

1. Откройте `.cursor/TEMPLATE_PROMPT_RU.md`.
2. Передайте этот prompt в Cursor.
3. Дождитесь, пока он обновит:
   - `.cursor/config.json`
   - `.cursor/rules/testing.mdc`
4. Только после этого начинайте использовать `/implement`, `/review` и другие команды.

Если `.cursor` уже давно используется в этом проекте и настройки уже подстроены под кодовую базу, этот шаг повторять не нужно.

## Что такое `.cursor`

В этой папке лежат настройки для Cursor:
- `commands/` - готовые slash-команды, которые можно вызывать в чате
- `agents/` - роли помощников, которых Cursor может подключать под задачу
- `rules/` - правила проекта: тесты, безопасность, документация
- `config.json` - базовые команды проекта: тесты, линтеры, форматирование, typecheck

Проще говоря: `.cursor` - это "инструкция для AI внутри проекта".

## Как пользоваться этим в реальной работе

Откройте чат Cursor внутри проекта и пишите команду прямо сообщением.

Базовый формат такой:

```text
/команда что нужно сделать
```

Примеры:

```text
/implement Добавь функцию нормализации номера телефона в `services/user_profile.py` и покрой тестами.
```

```text
/review Проверь мои изменения перед merge. Обрати внимание на обработку ошибок и регрессии.
```

```text
/refactor Упрости `handlers/payments.py` без изменения поведения. Если тестов не хватает, сначала скажи об этом.
```

Если нужно, можно сразу указывать файлы:

```text
/implement Измени `app/api/orders.py`: добавь фильтр по статусу и обнови тесты.
```

## Какую команду выбирать

### `/implement`
Используйте, когда нужно что-то реализовать:
- новая функция
- endpoint
- schema
- метод сервиса
- CLI-команда
- небольшая доработка существующего кода

Это команда "сделай задачу".

Когда подходит:
- задача локальная и понятная
- не нужно сначала долго проектировать решение
- изменение ограничено 1-3 частями системы

Примеры:

```text
/implement Добавь retry для запроса к внешнему API в `services/sync.py`.
```

```text
/implement В `bot/handlers/start.py` добавь проверку, что пользователь активен, и напиши тесты.
```

### `/orchestrate`
Используйте, когда задача большая или непонятно, с чего начать.

Это команда "разбей задачу на план и потом веди по шагам".

Когда подходит:
- новая фича целиком
- задача затрагивает много файлов
- есть миграции, интеграции, очереди, auth, billing
- хочется сначала получить план

Примеры:

```text
/orchestrate Нужно добавить импорт заказов из CSV с валидацией, логированием ошибок и фоновым запуском.
```

```text
/orchestrate Спланируй перевод авторизации на JWT и затем реализуй по шагам.
```

### `/refactor`
Используйте, когда надо улучшить код без изменения поведения.

Это команда "перепиши аккуратнее, но ничего не сломай".

Когда подходит:
- слишком большой метод
- дублирование
- плохая читаемость
- нужно разбить код на более понятные части

Важно: для безопасного рефакторинга желательно, чтобы уже были тесты.

Пример:

```text
/refactor Упрости `services/report_builder.py`, убери дублирование и сохрани текущее поведение.
```

### `/review`
Используйте для code review изменений.

Это команда "проверь, нет ли багов, рисков и дыр".

Когда подходит:
- перед merge
- после крупной серии изменений
- если хотите независимую проверку своего diff

Примеры:

```text
/review Проверь текущие изменения как перед merge в `main`.
```

```text
/review Проверь `api/auth.py` и `services/token_service.py` на баги и security-риски.
```

### `/audit`
Используйте, когда нужен обзор проекта целиком.

Это команда "сделай health check".

Что обычно смотрит:
- архитектуру
- безопасность
- качество кода
- тесты
- главные риски

Пример:

```text
/audit Проведи аудит проекта и дай список главных технических рисков и quick wins.
```

### `/write-tests-block`
Используйте, когда нужно написать тесты на конкретный кусок кода.

Можно указать:
- файл целиком
- или диапазон строк

Примеры:

```text
/write-tests-block Для `@services/payments.py:40-120` напиши pytest-тесты.
```

```text
/write-tests-block Для `@utils/dates.py` добавь минимально достаточный набор тестов.
```

## Что писать в запросе, чтобы результат был хорошим

Лучше всего работают запросы, где есть:
- что нужно сделать
- где именно это делать
- ограничения
- нужно ли писать тесты

Хороший шаблон:

```text
/implement Сделай <что нужно> в `<файл или модуль>`.
Ограничения: <что нельзя ломать / что важно сохранить>.
Нужно: <тесты / без тестов / сначала предложи план>.
```

Пример:

```text
/implement Добавь пагинацию в `api/users.py`.
Ограничения: не меняй формат текущего ответа, кроме добавления полей pagination.
Нужно: обновить pytest-тесты.
```

## Самый простой сценарий для новичка

Если вам дали задачу и вы не понимаете, с чего начать:

1. Если это первое использование `.cursor` в новом проекте - сначала актуализируйте шаблон через `.cursor/TEMPLATE_PROMPT_RU.md`.
2. Если задача маленькая и понятная - используйте `/implement`.
3. Если задача большая или расплывчатая - используйте `/orchestrate`.
4. После заметных изменений запустите `/review`.
5. Если менялась логика - просите тесты или используйте `/write-tests-block`.

Практически всегда можно начать так:

```text
/implement Вот задача: <вставьте текст задачи>. Сначала изучи нужные файлы, затем внеси изменения и обнови тесты.
```

Если боитесь, что задача слишком широкая:

```text
/orchestrate Вот задача: <вставьте текст задачи>. Сначала составь короткий план, потом выполняй по шагам.
```

## Какие проверки приняты в проекте

В проекте уже настроены базовые команды:
- `pytest -q`
- `pytest -vv`
- `flake8 .`
- `black .`
- `mypy .`
- `bandit -r . -x tests`

Ожидание проекта простое: новая или измененная бизнес-логика должна быть покрыта тестами или должно быть явно объяснено, почему тестов нет.

## Если совсем коротко

Запомните только это:
- `/implement` - сделать задачу
- `/orchestrate` - сначала разбить большую задачу на шаги
- `/refactor` - улучшить код без смены поведения
- `/review` - проверить изменения
- `/audit` - посмотреть на состояние проекта целиком
- `/write-tests-block` - написать тесты на конкретный код

Если вы новый человек в проекте и не уверены, что выбрать, начинайте с `/implement` для маленьких задач и с `/orchestrate` для больших.
